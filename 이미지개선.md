있지, 꽤 많아. Next 쓰면 사실 대부분 `next/image`가 알아서 해주는 게 커. 겹치지 않게 방법별로 정리해줄게.

**1. `next/image` 컴포넌트 쓰기 (가장 기본이자 핵심)**
`<img>` 대신 `next/image`를 쓰면 자동으로:
- **PNG를 WebP/AVIF로 변환**해서 서빙 (브라우저가 지원하면). PNG 그대로 안 내려보내고 훨씬 가벼운 포맷으로 바꿔줌 → 앞 대화에서 말한 "진짜 용량 줄이기"가 여기서 일어남
- 화면 크기에 맞춰 **자동 리사이즈** (srcset 생성)
- **lazy loading** 기본 적용 (뷰포트 밖 이미지는 나중에 로드)

```jsx
import Image from 'next/image'

<Image src="/hero.png" width={800} height={600} alt="..." />
```

**2. `priority`로 첫 화면 이미지 우선 로딩**
LCP(최대 콘텐츠 렌더링)에 잡히는 상단 히어로 이미지는 lazy load 하면 오히려 느려 보여. 이건 `priority` 붙여서 먼저 당겨오게 해.
```jsx
<Image src="/hero.png" priority ... />
```

**3. `placeholder="blur"`로 체감 속도 개선**
실제 로딩 속도는 아니지만 **체감**이 크게 좋아짐. 로딩 중 흐릿한 저해상도 미리보기를 깔아줌. 로컬 import 이미지는 blurDataURL 자동 생성.

**4. 사이즈 정확히 지정 + `sizes` 속성**
`width`/`height`를 명시하면 레이아웃 시프트(CLS)가 사라지고, 반응형이면 `sizes`로 실제 필요한 해상도만 받게 해서 과다 다운로드를 막아.
```jsx
<Image src="..." fill sizes="(max-width: 768px) 100vw, 50vw" />
```

**5. 애초에 PNG를 안 쓰기 (근본 해결)**
사진성 이미지면 PNG가 제일 무거운 포맷이야. 투명도 필요 없는 사진은 JPG/WebP, 아이콘·로고는 SVG가 훨씬 가벼워. PNG는 "투명도 있는 그래픽/스크린샷"일 때만 쓰는 게 맞아.

---

**추가로 알아둘 것**: `next/image`의 자동 변환은 서버에서 온디맨드로 이미지를 처리하는데, Vercel 배포면 이게 **Image Optimization 사용량으로 과금**돼. 트래픽 많으면 비용이 은근 나올 수 있어서, 안지향처럼 자체 인프라(AWS) 쓰는 경우엔 `loader`를 커스텀해서 CloudFront/S3 변환 파이프라인으로 빼거나, 빌드 타임에 미리 변환해두는 것도 방법이야.

**놓친 방향 하나**: 로딩 속도를 프론트 코드로만 풀려고 하기 쉬운데, PNG가 원본부터 무겁다면(예: 4000px 스크린샷을 그대로 올림) 그건 업로드/저장 단계 문제야. 앞서 말한 것처럼 업로드 훅에서 sharp로 미리 리사이즈·변환해두면 `next/image`가 처리할 원본 자체가 가벼워져서 서버 부하도, 변환 비용도 같이 줄어.
