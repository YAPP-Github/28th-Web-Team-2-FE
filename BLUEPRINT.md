# Agent 하네스 세팅 블루프린트

> 새 레포에서 agent 플로우를 세팅할 때 **하나씩 채워가는 템플릿**.
> `✍️ 채울 것:` 표시가 새 프로젝트에 맞춰 직접 적어야 하는 부분.
> 나머지(설계 원칙·구조)는 검증된 기본값이라 그대로 두고 시작하면 됨.

---

## 이 문서 사용법 (AI에게 주는 지시)

이 문서는 완성된 스펙이 아니라 **계속 채워가는 설계도**다. 완벽을 기다리지 말고, 채워진 만큼 만들고 빈 곳은 물어봐서 메워간다.

너(AI)가 이 문서를 받으면:

1. **먼저 §A 인터뷰 질문 뱅크로 사용자를 인터뷰한다. 질문을 아끼지 마라.** 카테고리마다 "이건 어떻게 되어 있나요? 저건요?"를 끝까지 묻는다. 모르는 건 추측으로 건너뛰지 않는다.
2. **한 카테고리씩** 진행한다. 답을 받을 때마다 **문서의 해당 칸을 즉시 갱신**한다 (대화가 끊겨도 문서에 남게).
3. 채워진 내용으로 `.claude/`를 구성한다. **도메인 정책·위험 경로·금지사항은 절대 지어내지 않는다** (환각 금지).
4. 사용자가 "모름/미정"이라 하면 `TODO(✍️): ____` 로 남기고 다음으로. **완벽을 기다리지 않는다.**
5. **agent 세팅도 인터뷰로 구체화한다(§A-2).** 기본 로스터를 출발점으로, 뺄지/합칠지/추가할지 사용자와 정한다.
6. 위험 경로(인증·결제 등)를 건드리는 산출물은 만들기 전에 사용자 확인(게이트)을 받는다.

## 산출물 구조 (Claude Code 기준)

- **agent** → `.claude/agents/<name>.md` — frontmatter(name/description/tools/model/skills/maxTurns/memory) + 본문에 그 agent의 작업 지침
- **스킬** → `.claude/skills/<name>/SKILL.md` — frontmatter(name/description) + 본문 매뉴얼. 규칙 모음은 `<name>/rules/` 하위에 **1파일 = 1규칙 + 탐지 패턴**
- **라우팅·최상위 불변 규칙** → 루트 `CLAUDE.md`

## 0. 용어 빠른 정의 (먼저 읽기)

| 용어 | 쉬운 뜻 |
|---|---|
| **agent** | 특정 역할만 맡는 일꾼. "탐색만", "구현만", "리뷰만" 식으로 분업. 각자 쓸 수 있는 도구·모델·매뉴얼이 정해져 있음 |
| **스킬(skill)** | agent한테 건네는 **참고 매뉴얼(마크다운 파일 1개)**. "우리 회사가 뭐 하는지", "API는 이렇게 짜라" 같은 지식 |
| **게이트(gate, ⏸)** | 작업을 자동으로 안 넘기고 **"여기서 멈추고 사용자 확인받는 정지선"**. 되돌리기 비싼 작업 직전에 둠 |
| **라우팅(routing)** | 들어온 요청을 **어느 agent/어느 절차로 보낼지** 정하는 분기 |
| **모델 티어(tier)** | 작업마다 다른 등급의 모델 배치 (가벼운 것 / 중간 / 똑똑한 것) |
| **escalation** | 막히거나 실패하면 **한 단계 위로 올리는 것** (모델을 올리거나, 사람한테 넘기거나) |

## 1. 설계 원칙 (개선안이 기본값)

### 1-1. 라우팅 — "크기 × 위험" 2차원

작업을 **파일 개수(크기)** 와 **건드리는 곳의 위험도** 두 축으로 본다.
(기존 세팅은 크기만 봐서, 위험한 1파일 수정이 검토 없이 지나가는 문제가 있었음 → 위험 축 추가)

```
            위험 낮음                위험 높음
          ──────────────        ────────────────────
크기 작음  │ S — 바로 진행      │ 게이트 + 리뷰 강제   │  ← 1파일이어도 위험경로면 절차 붙임
크기 큼    │ M — 탐색→구현→리뷰 │ L — 기획+게이트 풀절차│
```

- **크기**: 파일 1~2개 = 작음 / 3~5개 = 중간 / 5개+·새기능 = 큼
- **위험**: 아래 "위험 경로"를 건드리면 크기와 무관하게 한 단계 올림 + 게이트

> ✍️ 채울 것 — **위험 경로 목록** (건드리면 자동 escalate):
> - [ ] 인증/인가: `____`
> - [ ] 결제/돈: `____`
> - [ ] DB 스키마/마이그레이션: `____`
> - [ ] 공개 노출 페이지: `____`
> - [ ] (기타) `____`

> ✍️ 채울 것 — **자연어 → 플로우 신호**:
> - "바로/빨리" → 한 단계 내림 / "제대로/꼼꼼히" → 한 단계 올림
> - 단, **위험 경로는 신호로 못 내림** (말로 절차 스킵 방지)

### 1-2. 모델 티어 — "판단 밀도" 기준

똑똑한 모델을 **손이 많이 가는 일**이 아니라 **머리를 많이 써야 하는 일**에 배치한다.
(기존 세팅은 쓰기 권한 있는 구현 agent에 최상위 모델을 줬는데, 정작 미묘한 판단이 필요한 리뷰·설계가 약했음 → 판단 밀도 기준으로 교체)

| 판단 밀도 | 작업 성격 | 모델 티어 |
|---|---|---|
| **높음** | 모호함·설계·미묘한 버그 찾기 (리뷰, 기획, 버그조사) | 최상위 |
| **중간** | 명확한 패턴 따라 구현 | 중간 |
| **낮음** | 기계적 탐색·정리 | 가벼움 |

- **실패 기반 escalation**: 리뷰에서 2번 떨어지면 **수정은 한 단계 위 모델로**.
- **동적 override**: 가벼운 모델로 시작했는데 결론이 안 나면 한 단계 위로 재실행.

> ✍️ 채울 것 — 사용할 모델 3종 (가벼움/중간/최상위): `____ / ____ / ____`

### 1-3. 게이트 — "크기"가 아니라 "무엇을 바꿨나" 기준

빌드/검사 여부를 작업 크기가 아니라 **변경 내용**으로 정한다.
(기존: "작은 일=빌드 안 함" → 작은 일이 타입을 건드려도 검사 안 하는 구멍)

- **사용자 확인 게이트(⏸)를 둘 곳**: 기획 확정 직후 / 위험 경로 변경 직전 / 외부로 나가는 작업(배포·발행) 직전
- **자동 검사 게이트**: 타입·API·DB 변경이 있으면 크기 불문 타입체크/빌드

> 빌드/린트: `pnpm build`, `pnpm lint` / 유닛: `pnpm test` (Vitest) / E2E: Playwright
> ✍️ 채울 것 — 게이트를 둘 지점 목록 (위험 경로 정해지면): `____`

### 1-4. 단일 진실 소스 (어긋남 방지)

라우팅 안내표와 각 agent 설정이 **두 군데로 갈리면 어긋난다.**
→ 한 곳만 고치면 되게 하거나, 최소한 **"agent 추가/변경 시 같이 고칠 파일 체크리스트"** 를 둔다.

> ✍️ 채울 것 — agent 변경 시 동기화할 파일: `____`

## 2. SKILLS 정의 — "공통 1 + 전문 N"

매뉴얼을 **모두가 드는 공통 1개** + **필요한 agent만 집는 전문 N개**로 쪼갠다.

### 2-1. "1" — 공통 도메인 스킬 (전원 필수)

**프로젝트가 뭔지**를 담는다. 거의 모든 agent가 이걸 들고 시작한다.
아래 섹션이 **반드시 들어가야 할 골격** (anjihyang-domain 스킬 구조 기반):

- [ ] **제품 경계** — 한 코드베이스에 여러 제품/영역이면 라우트·권한·레이아웃 구분
- [ ] **비즈니스 & 수익 모델** — 뭘 팔고, 핵심 가치가 뭔지
- [ ] **사용자 단계** — 비회원 / 회원 / 유료 등 접근 권한 층
- [ ] **아키텍처 제약** — 인증 방식, 서버/클라 경계, 절대 못 하는 기술 제약
- [ ] **정책 금지사항** — 절대 하면 안 되는 것 (톤, 표현, 행위)
- [ ] **유입 채널** (있으면) — 어디서 사용자가 오는지
- [ ] **코딩 컨벤션 요약** — 세부는 전문 스킬로, 여기엔 최상위 원칙만
- [ ] **UI 원칙** (있으면) — 디자인 큰 방향

> ✍️ 채울 것: 위 항목을 새 프로젝트 내용으로. ← **"메인 도메인 정책 픽스"가 여기 들어감**

### 2-2. "N" — 전문 스킬 카탈로그

필요한 agent만 집어 든다. **거의 항상 필요 / 프로젝트마다 선택**으로 나눔.
각 스킬은 `이름 / 목적 / 들어갈 내용 / 누가 드나`.

#### A. 코드 컨벤션·패턴 (거의 항상 필요)

| 스킬 | 목적 / 들어갈 내용 | 주로 드는 agent |
|---|---|---|
| `[domain]-domain` | (위 §2-1, 공통 1) | 전원 |
| `api-patterns` | API 폴더 구조, 클라이언트, 에러 타입, 네이밍 규칙 | api/구현/리뷰 |
| `data-fetching` | 서버 상태 관리 패턴 (캐싱·무효화·낙관적 갱신) | api/구현 |
| `frontend-design` | 색·타이포·간격 토큰, 컴포넌트 규정 | UI구현/리뷰/디자인 |
| `form-patterns` | 폼 라이브러리 + 검증 스키마 조합 | 구현 |
| `coding-rules` | **rules/ 폴더: 1파일=1규칙 + 각 규칙에 탐지 패턴(Grep)** | 리뷰/구현 |

#### B. 이 프로젝트 특화 (프론트 전용 레포 + Figma + AI 테스트)

> 백엔드는 **외부 Spring(별도 레포)**. 이 레포엔 **백엔드 구현 agent 없음.** 백엔드 API를 *읽고* 프론트가 맞추는 역할만.

| 스킬 | 목적 / 들어갈 내용 | 주로 드는 agent |
|---|---|---|
| `figma-bridge` | Figma MCP 사용법, 노드→컴포넌트 매핑, **토큰은 Figma Variables → Tailwind config 자동 생성**(수동 입력 X, **매핑 안 되는 변수는 추측 말고 실패 처리**), **토큰 화이트리스트(config 밖 raw 값·arbitrary value `[13px]` 금지)**, MCP가 못 가져온 값은 **추측 금지 → 게이트**, 인증 실패 fallback | figma-implementer/design-reviewer |
| `backend-api-reference` | 외부 Spring API 문서/스펙 읽는 법, 엔드포인트·DTO 구조 → **프론트 API 레이어가 정확히 맞추도록**. **백엔드를 상상해서 만들지 말 것** | api-developer |
| `test-strategy` | AI-native 테스트: unit/E2E 범위, **"구현이 아니라 요구사항(스펙)을 검증"**, 동어반복 테스트 금지, 테스트 통과를 게이트로. E2E는 Playwright | test-writer/구현 |

## 3. AGENT 정의

### 3-1. agent 한 개의 설정 필드 (frontmatter)

```yaml
name:        agent 이름
description: 언제 쓰는지 + 인접 agent와의 경계 ("X는 이 agent, Y는 저 agent")
tools:       쓸 수 있는 도구 — 조사 agent는 Edit/Write 빼기 (읽기 전용)
model:       판단 밀도 기준 티어
skills:      들고 갈 매뉴얼 ("1" + 필요한 "N")
maxTurns:    최대 작업 턴 (폭주 방지)
memory:      project (작업 맥락 기억 필요한 agent만)
```

### 3-2. 권장 agent 로스터 (채워넣기)

읽기/쓰기 분리 + 판단 밀도 티어를 반영한 기본 구성.

> **프론트 전용 레포.** 백엔드 구현 agent 없음 — `api-developer`가 외부 Spring 스펙을 *읽고* 맞춤.

| agent | 역할 | 읽기/쓰기 | 티어 | 든 스킬 |
|---|---|---|---|---|
| explorer | 빠른 코드 탐색 | 읽기 | 가벼움 | domain |
| auditor | 누락 없는 전수 검색 | 읽기 | 중간 | domain |
| planner | 기획·기술 결정·계획 | 읽기 | **최상위** | domain |
| flow-reviewer | 유저 플로우 검수 (CRUD·사용자 관점 누락) | 읽기 | **최상위** | domain, flow-review |
| bug-investigator | 버그 근본원인 추적 (수정X) | 읽기 | **최상위** | domain, api |
| api-developer | 프론트 API 레이어 (외부 Spring 스펙 보고 맞춤) | 쓰기 | 중간 | domain, api, backend-api-reference, data-fetching |
| frontend-dev | 프론트 화면/로직 구현 | 쓰기 | 중간 | domain, api, design, form |
| wireframe-builder | 디자인 전 와이어프레임 초안 (더미 데이터·배포) | 쓰기 | 중간 | domain, wireframe-drafting, form |
| figma-implementer | Figma→코드 변환 (토큰 화이트리스트 준수) | 쓰기 | 중간 | domain, figma-bridge, design |
| test-writer | AI-native 테스트 작성·실행·수정 | 쓰기 | 중간 | domain, test-strategy |
| code-reviewer | 코드 리뷰 | 읽기 | **최상위** | domain, rules, api, design |
| design-reviewer | Figma 스펙 일치 + 토큰 위반 검토 | 읽기 | 중간~상위 | domain, figma-bridge, design |
| diff-organizer | 커밋 정리 | 쓰기(git) | 가벼움 | — |

> ✍️ 채울 것 — 위에서 **실제로 둘 agent 확정** + 각자 든 스킬을 §2 확정 목록과 맞추기.
> (디자이너 워크플로우에 따라 화면구현을 frontend-dev에 합칠지 figma-implementer로 뗄지 결정)

## 4. 플로우 정의

### 4-1. 일반 (크기 × 위험 라우팅)

```
S  │ 기존코드 확인 → 변경요약 1줄 → 구현            (위험경로 아니면 검사 생략 가능)
M  │ explorer → 구현 → reviewer ⟳(최대2회)
L  │ planner ║ explorer → [⏸ 확인] → 구현 → reviewer + 빌드 ⟳ → (요청시) diff-organizer
   │ ※ 위험경로 변경 → 크기 무관 [⏸] + reviewer 강제
```
( `║`=병렬, `⏸`=게이트, `⟳`=루프 )

### 4-2. 전용 플로우 (트리거별)

> ✍️ 채울 것 — 새 프로젝트에 맞춰 확정. 후보:

```
신규 화면 │ requirements → [⏸] → (data-model) → figma 참조 구현 → design-reviewer + code-reviewer ⟳
          │   ※ "Figma 확정됨?" 게이트, MCP 안되면 fallback
풀스택    │ planner ║ explorer → [⏸] → frontend-dev ║ backend-dev → reviewer + 빌드/테스트 ⟳
          │   ※ 스키마 변경 [⏸]
Bug       │ bug-investigator(수정X) → frontend|backend-dev → reviewer
전수검색  │ auditor(누락0) → 구현(일괄) → reviewer
```

## A. 인터뷰 질문 뱅크 (AI → 사용자)

> 받는 AI는 이 질문들로 사용자를 **적극적으로 인터뷰**한다. 한 카테고리 끝낼 때마다 문서 갱신.
> 🔴 = AI가 절대 못 지어냄(반드시 사용자에게) / 🟡 = 기본값 있으나 확인 필요.
> 사용자가 한 번에 답을 다 못 줘도 됨 — 모르면 `TODO`로 두고 다음 카테고리로.

### A-1. 도메인 인터뷰 (→ `[domain]-domain` 스킬로 정리)

**0. 제품 경계** 🔴
- 한 코드베이스/레포 안에 여러 제품·영역이 있나요? (예: 공개 사이트 + 로그인 전용 공간)
- 라우트로 어떻게 나뉘죠? 각 영역의 **대상·접근 권한·목표·레이아웃**이 어떻게 다른가요?
- 영역 간 컴포넌트를 공유하면 안 되는 선이 있나요?

**1. 비즈니스 & 서비스** 🔴
- 무엇을 파나요? **메인 수익 모델**은?
- 핵심 가치를 한 문장으로?
- **절대 하면 안 되는 것**(말투·표현·행위·법적 제약)은?
- 사용자 단계(비회원 / 회원 / 유료 등)와 각 단계 권한은?
- 유입 채널은 어디(검색·SNS·광고·추천)?
- 향후 로드맵/방향은?

**2. 핵심 구조** 🟡
- 메인 네비게이션은 어떻게 구성되나요?
- 홈 화면의 역할은? (랜딩 / 대시보드 / 피드)
- 핵심 데이터/콘텐츠의 계층 구조는?
- 권한(role) 체계는?
- 인증 방식은? (httpOnly 쿠키 / 토큰 / 소셜 로그인)

**3. 아키텍처 제약** 🔴 ← *미정인 "렌더링 전략"이 여기*
- 인증 방식이 SSR/Server Component를 제약하나요?
- 페이지 기본은 **Server냐 Client냐**? 판단 기준은?
- **BFF를 둘 건가요?** API Route는 어떤 용도로만 쓰나요?
- SEO/메타데이터 전략은? (정적 / 동적 / generateMetadata)

**4. 코딩 컨벤션** 🟡 (대부분 전문 스킬에 있음, 최상위 원칙만)
- 절대 규칙(any 금지 / barrel export 금지 등)은?
- 상태 관리 도구와 선택 기준은?
- 네이밍 규칙(API 훅, 파일)은?

**5. UI 원칙** 🟡
- 타겟 사용자 특성은? (연령대 / 주 사용 디바이스 / 접근성 요구)
- 디자인 시스템 출처는? (Figma — 토큰 매핑 규칙은 §A-2)
- 모바일 퍼스트인가요?

**6. 개발 환경** 🟢
- 배포(Vercel?) / 모니터링(Sentry?) / 분석(GA4?)은?
- 브랜치 전략과 릴리즈 방식은?

### A-2. 하네스 · agent 인터뷰 (→ agent 파일 + CLAUDE.md로 정리)

**위험 & 게이트** 🔴
- 어떤 폴더/기능을 건드리면 위험한가요? (인증·결제·개인정보 등 → 자동 escalate 경로)
- 작업을 자동으로 끝까지 가도 되는 지점 vs **반드시 멈추고 확인받을 지점**은 어디?

**agent 로스터** 🟡 (§3-2 기본 로스터를 보여주고)
- 이 기본 구성에서 **뺄 것 / 합칠 것 / 추가할 것**이 있나요?
- 화면 구현을 `frontend-dev`에 합칠까요, `figma-implementer`로 뗄까요?
- 테스트 전담 `test-writer`를 둘까요, 구현 agent가 겸할까요?

**모델 티어** 🟡
- 가벼움 / 중간 / 최상위에 각각 어떤 모델을 쓸까요? (판단 밀도 기준)
- 리뷰·기획처럼 머리 쓰는 곳에 상위 모델을 배치하는 데 동의하시나요?

**테스트 강도** 🟡
- "빡세게"의 범위는? 위험 경로는 **테스트 통과를 머지 게이트로** 강제할까요?
- E2E(Playwright)를 어느 흐름까지 커버할까요?

**Figma 토큰 자동화** 🔴
- Figma Variables **이름 → 코드 토큰 이름** 매핑 규칙은? (예: `color/brand/500` → `brand-500`)
- 매핑 안 되는 변수가 나오면 실패 처리하는 데 동의하시나요? (상상코딩 차단)

**운영** 🟢
- diff/커밋 정리를 자동화할까요?
- agent/스킬 추가 시 어느 파일을 같이 갱신할지(단일 진실 소스)?
