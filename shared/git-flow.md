# Git 워크플로우 (단일 진실 소스 · 도구 무관)

> diff-organizer / PR 작업이 따른다. **공동작업자 있음** 전제 — 공유 브랜치 안전이 최우선.

## 커밋·푸시 요청 시 — 먼저 질문 (필수 · override)

- 사용자가 "커밋/푸시 해줘"라고 하면 **agent가 멋대로 브랜치를 따지 않는다.** 임의 브랜치 생성으로 히스토리를 꼬이게 하는 것 금지.
- **반드시 먼저 한 가지를 묻는다 — 둘 중 무엇인지**:
  1. **현재(로컬) 브랜치에 그대로 커밋**(필요 시 그 브랜치만 푸시), 또는
  2. **새 작업 브랜치를 따서 PR**로 올리기.
- 사용자가 이미 방식을 명시한 경우에만 그대로 진행. 불명확하면 추측 금지 — 위 질문으로 확인하고 진행.
- 이 질문은 아래 "게이트(커밋·푸시 직전)"와 함께 매번 적용된다.

## 브랜치

- 새 작업 브랜치 생성은 **위 질문에서 "브랜치+PR"을 택했을 때만**. (자동으로 따지 않음)
- 공유 루트(`main`/`dev`)에 **직접 푸시·force push 금지**. 루트는 **PR merge로만** 갱신.
- 네이밍: `feat/<짧은설명>` `fix/<…>` `refactor/<…>` (팀 공통 규격)

## 커밋

- 형식: `feat(scope): 한국어 설명` (feat/fix/refactor/chore/style/docs)
- 논리 단위로 분리.

## PR 직전 흐름 (그대로 실행)

```
git switch feat/<…>
git fetch origin
git rebase origin/dev            # 루트 최신으로 rebase
# ⚠ 충돌 → agent는 자동 해결 금지. "어디를 어떻게 합칠지" 반드시 사용자에게 묻는다
git push --force-with-lease      # 내 작업 브랜치만 (--force 금지)
→ PR 생성
```

## 게이트 (⏸ 사용자 확인)

- 커밋·푸시 직전 → **"로컬 그대로 vs 브랜치+PR" 먼저 질문**(위 섹션)
- 위험 경로(인증·결제 등) 변경 직전
- 배포 직전
- rebase 충돌 발생 시

## PR 코멘트 규격

- 리뷰 출력은 `review-standard.md` 고정 템플릿. 팀 전원·모든 도구 동일 형식.
